# План MVP: iPhone-приложение для борьбы с джетлагом

## Концепция

Приложение для iPhone, которое строит **персональное расписание световых воздействий, приёма мелатонина и сна** на основе конкретного перелёта пользователя. Цель — помочь организму перестроиться к новому часовому поясу быстрее и мягче, опираясь на принципы хронобиологии.

**Главный принцип MVP:** одна основная ценность, сделанная хорошо.

---

## Что входит в MVP

### Включаем
- Ввод данных о перелёте (откуда, куда, когда вылет/прилёт)
- Ввод обычного режима сна пользователя
- Расчёт расписания на 2–3 дня до, во время и 2–3 дня после полёта
- Таймлайн с конкретными действиями:
  - «Искать свет» / «Избегать света»
  - «Принять мелатонин»
  - «Спать» / «Просыпаться»
  - «Кофеин можно / нельзя»
- Push-уведомления в нужные моменты

### Откладываем на потом
- Интеграция с Apple Health / HealthKit
- Поддержка нескольких городов в одной поездке
- Социальные функции, шеринг
- AI-чат, персональный коуч
- Подписочная модель с тиерами (сначала проверить, работает ли продукт)

## Архитектура алгоритма

Это сердце приложения. От качества алгоритма зависит, будет ли продукт реально работать или станет «ещё одним напоминальником».

### Шаг 1. Определить текущую циркадную фазу

Самый надёжный прокси без датчиков — **Midsleep Point** (середина сна).

Пример: если человек ложится в 23:00 и встаёт в 7:00, его midsleep = 3:00.

Минимум температуры тела (**CBTmin**) обычно происходит за 2–3 часа до пробуждения — около 4–5 утра. Это критическая точка, относительно которой работает вся Phase Response Curve.

### Шаг 2. Определить целевую фазу

В новом часовом поясе midsleep должен сместиться так, чтобы человек комфортно спал ночью по местному времени.

### Шаг 3. Выбрать направление сдвига

- **Восток (Москва → Бали, +5 часов)** → сдвиг фазы вперёд (advance)
- **Запад** → сдвиг фазы назад (delay)

Сдвиг назад организму даётся легче, так как внутренние часы естественно длиннее 24 часов. При разнице больше 8–10 часов иногда эффективнее сдвигать в «неочевидную» сторону.

### Шаг 4. Применить Phase Response Curve (PRC)

Ключевая научная основа. Упрощённая версия:

| Воздействие | Время | Эффект |
|---|---|---|
| Свет | За 2–3 часа до CBTmin | Сдвиг назад (delay) |
| Свет | Через 2–3 часа после CBTmin | Сдвиг вперёд (advance) |
| Мелатонин | Вечером (~5 часов до сна) | Сдвиг вперёд |
| Мелатонин | Утром | Сдвиг назад |

**Важно:** алгоритм не должен сдвигать фазу больше чем на 1–1.5 часа в день — иначе организм не успевает адаптироваться.

## Технический стек

Для iPhone MVP рекомендуется:

- **SwiftUI + Swift** — нативная разработка, быстрый UI, отличные анимации для таймлайна
- **Локальные вычисления** — алгоритм PRC помещается в одной Swift-функции, серверная часть в MVP не нужна
- **UserNotifications framework** — для push-уведомлений в нужное время
- **Core Data или JSON в FileManager** — для хранения планов поездок
- **TimeZone API** — встроенная работа с часовыми поясами

**Бэкенд в MVP не нужен.** Все расчёты делаются на устройстве. Это упрощает:
- Запуск
- GDPR-комплаенс
- Затраты (нет расходов на серверы)

## Минимальный набор экранов

1. **Onboarding** — пара вопросов про обычный режим сна
2. **Новая поездка** — форма с данными о перелёте
3. **План** — главный экран с таймлайном по дням
4. **Текущее действие** — что делать прямо сейчас + таймер до следующего события
5. **Настройки** — режим сна, разрешения на уведомления

## Что критически важно проверить перед запуском

### Научная корректность
Найти хронобиолога-консультанта или серьёзно изучить работы:
- Charles Czeisler (Гарвард)
- Helen Burgess
- Charmane Eastman

Неправильный алгоритм может реально ухудшить состояние пользователя — например, посоветовать свет в момент, когда он сдвинет фазу в противоположную сторону.

### Дисклеймеры о мелатонине
Статус мелатонина различается по странам:
- **Россия** — рецептурный в некоторых дозировках
- **США** — БАД, продаётся свободно
- **ЕС** — варьируется

Приложение должно рекомендовать консультацию с врачом перед использованием.

### Точность часовых поясов
Классический источник багов:
- Летнее время (DST)
- Часовые пояса с получасовым сдвигом (Индия, части Австралии)
- Пересечение линии перемены дат
- Изменения часовых поясов в отдельных странах

## План разработки по шагам

1. **Прототип алгоритма** в Python или Swift Playground — функция, которая на вход получает перелёт, на выход даёт массив событий. Без UI.
2. **Валидация алгоритма** — сравнить выводы с рекомендациями Timeshifter на одинаковых маршрутах.
3. **UI-прототип в Figma** — продумать визуализацию таймлайна.
4. **SwiftUI-приложение** — собрать MVP за 2–4 недели.
5. **TestFlight** с 20–50 путешественниками — сбор обратной связи, особенно про точность рекомендаций.

## Конкуренты для анализа

- **Timeshifter** — лидер рынка, основан на работах Czeisler
- **Flykitt** — комплексный подход (приложение + добавки)
- **StopJetLag** — более старый игрок

Стоит протестировать каждое на одинаковом маршруте (например, Москва → Бали) и проанализировать различия в рекомендациях.
